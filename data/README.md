# Data

This folder stores the input, intermediate, and generated data used by the Tranco DNS dependency study. It may be empty, or nearly empty, in the public repository because many parquet files are large and are generated by the notebooks in `code/`.

To reproduce the study, either run the notebooks in the order described in [`../code/README.md`](../code/README.md), or download the prepared data archive from the project site and unzip it so the folders below appear under `data/`.

## Folder Structure

### `data/names/`

Domain and TLD metadata used by the modeling and analysis notebooks.

Typical files:

- `tranco_categorized_names.parquet`: Tranco domains with rank, TLD, TLD type, and parent-status metadata. Generated by `ptt-iyp-modeling_dependency_tranco_list.ipynb`.
- `tranco_unresolved_domains_by_type.parquet`: unresolved domains grouped by TLD type. Generated by `ptt-iyp-analysis_(tranco).ipynb`.
- `ccTLD_registration_date_may_10_2026.parquet`: ccTLD registration dates used for old-TLD analysis. Generated or reused by `ptt-iyp-analysis_(tranco).ipynb`.

### `data/result/dns_dependency/`

Core DNS dependency outputs. These files connect Tranco domains to the authoritative nameserver dependency sets used to compute TCB.

Typical files:

- `ccTLD_domains_dependency_no_parent.parquet`
- `gTLD_domains_dependency_no_parent.parquet`
- `gen_resTLD_domains_dependency_no_parent.parquet`
- `sponTLD_domains_dependency_no_parent.parquet`
- `infrTLD_domains_dependency_no_parent.parquet`
- `special_use_domains_dependency_no_parent.parquet`
- `domain_with_parent_rel.parquet`
- `IP_Prefix_AS_dependency_sample.parquet`

The domain dependency files are generated by `ptt-iyp-modeling_dependency_tranco_list.ipynb`. The IP/prefix/AS sample is generated by `ptt-iyp-analysis_(tranco)_infrastructure_ip_prefix_AS.ipynb`.

### `data/result/NS_infrastructure/`

Nameserver service-provider summaries used to measure dependency on dominant DNS infrastructure providers.

Typical file:

- `tranco_NS_Infrastructure.parquet`: generated by `ptt-iyp-analysis (tranco)_infrastructure_NS_service_provider.ipynb` and consumed by the final analysis notebook.

Note: the current nameserver infrastructure notebook writes this file first to `data/result/dns_dependency/tranco_NS_Infrastructure.parquet`. The final analysis expects it under `data/result/NS_infrastructure/`, so keep a copy there or update the analysis path cell locally.

### `data/result/IP_Prefix_AS_dependency/`

Chunked full output from the IP, BGP prefix, and autonomous-system dependency mapping notebook.

Typical files:

- `chunk_*.parquet`

These files are generated by `ptt-iyp-analysis_(tranco)_infrastructure_ip_prefix_AS.ipynb` and consumed by the final analysis notebook for AS and prefix dependency figures.

### `data/anycast_census/`

Downloaded Anycast Census/LACeS snapshots and high-confidence anycast subsets.

Typical files:

- `laces_ipv4_*.parquet`
- `laces_ipv6_*.parquet`
- `laces_ipv4_anycast_high_*.parquet`
- `laces_ipv6_anycast_high_*.parquet`

These files are generated or reused by `ptt-iyp-analysis_(tranco)_infrastructure_IP_anycast.ipynb`.

### `data/result/anycast_results/`

Per-domain anycast summaries derived from the IP dependency data and Anycast Census snapshots.

Typical file:

- `domain_anycast_summary.parquet`: generated by `ptt-iyp-analysis_(tranco)_infrastructure_IP_anycast.ipynb` and consumed by the final analysis notebook.

### `data/result/temp/`

Reusable intermediate analysis tables.

Typical file:

- `resolved_summary.parquet`: generated by the final analysis notebook after combining resolved dependency data. It lets later analysis cells resume without recomputing earlier preprocessing.

### `data/result/without_allias/`

Earlier or comparison dependency outputs without alias handling. These files are useful for methodological comparison but are not the main Tranco workflow described in the current code README.

## Restoring Data From an Archive

If a prepared data archive is provided on the project site:

1. Download the archive.
2. Unzip it into the repository root or directly into `data/`.
3. Confirm that paths such as `data/names/`, `data/result/dns_dependency/`, and `data/result/NS_infrastructure/` exist.
4. Start the final or downstream notebooks only after their required input files are present.

If a notebook reports a missing parquet file, check the file path in the error and rerun the notebook that generates that file.
